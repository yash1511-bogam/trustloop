# Webhook Integrations

TrustLoop supports six webhook integrations:

- Datadog: `/api/webhooks/datadog`
- PagerDuty: `/api/webhooks/pagerduty`
- Sentry: `/api/webhooks/sentry`
- Langfuse: `/api/webhooks/langfuse`
- Helicone: `/api/webhooks/helicone`
- Generic: `/api/webhooks/generic`

## Authentication Modes

Webhook routes accept either:

1. Workspace API key auth (`Authorization: Bearer sk-tl-...`)
2. Signed secret headers (`x-trustloop-workspace`, `x-trustloop-signature`, `x-trustloop-timestamp`)

Signed secret mode is recommended for external systems.

## Setup Steps

1. Open **Settings -> Integrations**.
2. Choose integration type and save secret.
3. Copy endpoint path and configure your source system.
4. Send required headers with each request.

## Signature Format

TrustLoop verifies `HMAC-SHA256` signatures.

- If timestamp is present, sign `"<timestamp>.<raw_body>"`
- If timestamp is missing, sign `<raw_body>`
- Header formats accepted: raw hex, `sha256=<hex>`, or `v1=<hex>`
- Signature timestamp window: +/- 300 seconds

## Example Request (Signed Secret)

```bash
curl -X POST "https://your-app.com/api/webhooks/generic" \
  -H "Content-Type: application/json" \
  -H "x-trustloop-workspace: workspace-slug-or-id" \
  -H "x-trustloop-timestamp: 1762675200" \
  -H "x-trustloop-signature: sha256=<hmac_hex>" \
  -d '{
    "title": "Spike in hallucinations",
    "description": "Customer-facing wrong answers crossed SLA threshold",
    "severity": "P1",
    "sourceRef": "obs-incident-420"
  }'
```

## Response Behavior

- `201`: incident created
- `200`: deduplicated by `sourceTicketRef` and existing incident returned
- `400`: invalid JSON object or invalid mapped payload
- `401`: auth/signature failed
- `429`: incident quota reached

## Deduplication

If a mapped payload includes `sourceTicketRef` (or equivalent), TrustLoop deduplicates within workspace scope and returns the existing incident instead of creating a duplicate.

## Operations Notes

- Post-create side effects (quota consume and read-model refresh) are non-blocking and logged.
- Every webhook lifecycle stage is now structured-logged for traceability.
